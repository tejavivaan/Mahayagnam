# kill Command 
## What is a Process?
- A process is any running program in Linux.
- Examples: Firefox, Apache, Python script, shell script, background job, etc.
- Every process has a PID (Process ID) — a unique number.
- To see processes: ps -ef, top, htop

## What is the kill Command?
- kill is a command used to send signals to processes.
- It does NOT always terminate a process — it depends on the signal you send.
- kill = send a signal
- signal = instruction to the process
- Examples of instructions: Stop, Continue, Terminate, Restart, Reload configuration.

## Why Do We Need kill?
- A process is stuck or frozen
- A service needs to be restarted
- A background job must be stopped
- A script is consuming too much CPU
- A zombie process needs cleanup
- A process must be gracefully shut down

## Structure of the kill Command
- ***kill [signal] PID***
## Understanding Signals 
- Linux has 64 signals.
- You can list them: ***kill -l***
## How kill Works Internally (Beginner Friendly)
- You send a signal
- Kernel receives it
- Kernel delivers it to the process
- Process decides:
  - Handle it
  - Ignore it
  - Terminate
# Process Signals in Linux
## What Are Signals in Linux?
- Signals are software interrupts sent to a process to notify it that an event has occurred.
- Think of signals as messages from the kernel or another process.
- Examples of events:
  - User pressed Ctrl+C
  - A process must stop or terminate
  - A child process exited
  - A configuration reload is needed
  - A timer expired
  - A segmentation fault occurred
- Signals allow Linux to control processes without directly interfering with their memory or execution.

## Why Do Signals Exist?
- Signals help Linux:
  - Stop or kill processes
  - Pause and resume processes
  - Handle errors (segfault, illegal instruction)
  - Communicate between parent and child processes
  - Reload configuration files
  - Gracefully shut down services

## How Signals Work Internally (Beginner Friendly)
- A signal is generated By kernel, By user (Ctrl+C), By another process (kill command)
- Kernel delivers the signal to the target process
- Process decides:
  - Handle the signal (custom handler)
  - Ignore the signal (if allowed)
  - Terminate immediately
- Some signals cannot be ignored (e.g., SIGKILL, SIGSTOP).
- How to View All Signals
  - ***kill -l***
  - This lists all 64 standard Linux signals.
## Most Important Linux Signals
### SIGTERM (15) — Graceful Termination
- Default signal for kill
- Allows cleanup
- Saves data
- Recommended way to stop processes
- ***kill -15 PID*** or ***kill PID***
### SIGKILL (9) — Force Kill
- Cannot be ignored
- Cannot be handled
- Cannot be blocked
- Immediate termination
- ***kill -9 PID***
- Use only when SIGTERM fails.
### SIGINT (2) — Interrupt
- Triggered by Ctrl+C
- Stops a foreground process
- ***kill -2 PID***
### SIGQUIT (3) — Quit + Core Dump
- Triggered by Ctrl+\\
- Used for debugging
- ***kill -3 PID***
### SIGHUP (1) — Hangup / Reload Config
- Originally: terminal disconnected
- Now: reload configuration for daemons
- ***kill -1 PID***
### SIGSTOP — Pause Process
- Cannot be ignored
- Kernel-level stop
- ***kill -STOP PID***
### SIGCONT — Resume Process
- ***kill -CONT PID***
### SIGCHLD — Child Process Exited
- Sent to parent when: Child exits, Child stops, Child resumes
- Used by init/systemd to clean zombies.
### SIGSEGV — Segmentation Fault
- Occurs when a process accesses invalid memory.

### SIGALRM — Timer Expired
- Used by alarms and timeouts.

### SIGUSR1 & SIGUSR2 — User-Defined Signals
- Used by applications for custom actions.
