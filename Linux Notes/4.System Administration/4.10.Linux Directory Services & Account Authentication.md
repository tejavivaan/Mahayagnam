# Linux Directory Services & Account Authentication

## What Are Directory Services & Authentication in Linux?

- Directory Service
  - A centralized database that stores: Users, Groups, Computers, Policies, Access rules.
- Authentication
  - Verifying who the user is.
- Authorization
  - Deciding what the user can do.

- **Linux supports**:
  - Local authentication
  - Centralized authentication (LDAP, AD, FreeIPA)

- **Local User Authentication (Default in Linux)**
  - Linux stores users locally in:
    - /etc/passwd → user info
    - /etc/shadow → encrypted passwords
    - /etc/group → group info
  - Local users are good for: root, service accounts, small labs
  - But not scalable for enterprises.

- **Why Enterprises Need Directory Services**
  - When companies have:
    - 500+ Linux servers
    - 50+ sysadmins
    - Strict security policies
  - Local users become a nightmare.
  - Problems with local users:
    - Must create/delete users on every server
    - Password changes must be done everywhere
    - No centralized audit
    - High security risk
  - Solution → Directory Services

## What Is LDAP? 
- **LDAP = Lightweight Directory Access Protocol**
- It is NOT a directory service
- It is a protocol used to query and modify directory data
- Think of LDAP like:
  - HTTP → used to access websites
  - LDAP → used to access directory services
- Directory services that use LDAP:
  - Active Directory, OpenLDAP, FreeIPA, IBM Directory Server, JumpCloud (LDAP-as-a-service).
- LDAP is used for:
  - User lookup, Group lookup, Authentication, Authorization.

## Some Directory Services
### Active Directory (AD)
- Active Directory = Microsoft’s Directory Service
- It is the most widely used directory service in the world.
- **Key Features**
  - Centralized user management
  - Group Policies (GPOs)
  - Kerberos authentication
  - LDAP support
  - DNS integration
  - Computer management
  - Organizational Units (OUs)
- **Where AD is used?**
  - 99% of enterprises
  - Windows-heavy environments
  - Hybrid cloud setups
- **AD Authentication Flow**
  - User enters username/password
  - Linux/Windows sends request to AD
  - AD validates using Kerberos
  - LDAP used for user lookup
  - Access granted

### OpenLDAP
- OpenLDAP = Open-source directory service
- Runs mostly on Linux/Unix.
- **Key Features**
  - Lightweight
  - Highly customizable
  - Uses LDAP protocol
  - No Kerberos by default
  - Used in Linux-heavy environments
- **Where OpenLDAP is used?**
  - Universities
  - Legacy Linux systems
  - Lightweight authentication setups

### IBM Directory Server
- IBM’s enterprise-grade directory service.
- **Key Features**
  - LDAP-based
  - Highly scalable
  - Used in banking, telecom, government
  - Supports multi-master replication
  - Integrates with IBM security products
- **Where it is used?**
  - Large enterprises with IBM stack
  - High-security environments

### JumpCloud
- JumpCloud = Cloud-based Directory Service (Directory-as-a-Service)
- **Key Features**
  - Cloud-hosted LDAP
  - Cloud-hosted RADIUS
  - Zero-trust access
  - Device management (Windows, Mac, Linux)
  - SSO + MFA
  - No on-prem servers needed
- **Where JumpCloud is used?**
  - Startups
  - Cloud-native companies
  - Remote-first organizations

### Identity Manager (IDM)
- IDM = Identity Management System
- It is NOT a directory service.
- It is a management layer on top of directory services.
- Examples: Red Hat IDM (FreeIPA), Oracle Identity Manager, SailPoint, Okta Identity Manager
- **What IDM does?**
  - Provisioning users
  - De-provisioning users
  - Password policies
  - Access workflows
  - Role-based access control
  - Integration with AD/OpenLDAP
  - IDM = Automation + Governance
  - Directory Service = Storage + Authentication

### Winbind
- Winbind is part of Samba.
- **What Winbind does?**
  - Connects Linux machines to Active Directory
  - Allows Linux to authenticate using AD users
  - Maps Windows SIDs to Linux UIDs
  - Works with Kerberos + LDAP
- **Where Winbind is used?**
  - Mixed environments (Windows + Linux)
  - Samba file servers
  - Linux servers joining AD domain
### Differences Between All of Them
- | Technology | Type | Uses LDAP? | Purpose | Best For |
  | --- | --- | --- | --- | --- |
  | **[LDAP](ca://s?q=Explain_LDAP)** | Protocol | Yes | Query directory data | All directory services |
  | **[Active Directory](ca://s?q=Explain_Active_Directory)** | Directory Service | Yes | Enterprise authentication | Windows-heavy companies |
  | **[OpenLDAP](ca://s?q=Explain_OpenLDAP)** | Directory Service | Yes | Lightweight directory | Linux-heavy setups |
  | **IBM Directory** | Directory Service | Yes | Enterprise-grade LDAP | Banking, telecom |
  | **JumpCloud** | Cloud Directory | Yes | Cloud identity | Cloud-native companies |
  | **IDM** | Identity Manager | Uses LDAP indirectly | User lifecycle automation | Large enterprises |
  | **Winbind** | Integration Tool | Uses LDAP + Kerberos | Connect Linux to AD | Mixed OS environments |
